Zombie Tutorial

Week 1

First week is about setting up the project. For the course of this tutorial I will be using Unity 2019.2 and only Standard or free assets from the assets store. I opened a new empty project and imported the Standard Assets. What I didn’t take into consideration is that I only need the characters from this library so I could have imported those instead of the whole package.

The next step was to place my player and the AI in the scene. It is very curious that the player movement is affected by the main camera. Although the camera is attached to the character and untagged, the movement will not be the desired one. To fix both the movement and the camera clipping problem I have opted for a top-down view and made a script which tells the camera to follow the player.

Following the tutorial, I had to place some obstacles for the AI to avoid. For this step I wanted to make a level a bit more elaborate. I used a maze generator imported from the Asset Store in order to create the environment. Because the AI paths needs to be baked into the NavMesh and the maze is procedurally generated I couldn’t use it in this form, so I played the game and made a prefab out of the maze. Then I’ve made some small adjustments to my liking and baked the NavMesh.

The AI character needs a destination to go towards now, so the next step is to create a script that sets this target. At this stage in the tutorial the user controls where the AI goes which reminds me of the game mechanics in real-time strategy games such as Warcraft III, Starcraft, Dota. I intend to use this control for controlling the player.

Week 2

This week’s mission is to build a Finite State Machine for the zombie. By the end of this week the enemy will be able to Chase, Wander and Die. I used an enum to consider the different the AI can be in. I placed a few Waypoints around the map so the zombie can pick and move towards the closest one. These points are stored in an array and shuffled so each time there is a unique path for the AI.

The next step is to change the default character, Ethan, with something more relevant. The tutorial suggests using Mixamo, however, I wanted to add more features to my game design, so I have chosen three characters from the Asset Store. One is the player, a Sci-fi robo-solider, then the zombie and, finally, the aid of the player which is a drone controlled by AI.

Week 3

In this week the zombie will receive sight and he’ll be able to see the player. The way we do that is by performing a raycast inside the Field of View of the AI. If the ray hits an object and the object is the player, the we return true and we change the state from WANDERING to CHASING.

Following the tutorial, I have added a DEATH state and mechanics. I have also added improvements to the game design, for example, a health system for the enemy and the player, enemy attack and player shooting, and now, both the player and the enemy can kill each other.


Week 4

In the final week, I have to create a gameplay for this AI project and the way suggested in the tutorial is by spawning a number of zombies at one of the Waypoints placed on the map and keep spawning them when they die. I disagree with the code in the tutorial because every frame it performs a search in order to count how many zombies are alive. It can be argued that for the purpose of this tutorial and the number of assets in the scene it works very well, however, it is bad practice to write inefficient code. My solution is to create a static variable called zombiesAlive ( aka nrZombiesExist ) and control the current number of zombies by subtracting 1 when it dies and adding 1 when a new enemy is spawned. This way I can remove the search completely because I know the number of zombies alive at any given time.

To further improve the performance of the game, I want to make use of Coroutines and add a Pooling system for the zombies. Coroutines help decrease the frequency of tasks performed which makes the game run faster with no drawbacks of making the AI less aware. For example, we can check if the zombie sees the player twice a second instead of 120 times a second and there would be no difference. Because creating new objects is a very expensive task for the CPU we can have a boost the performance by using a pooling system. It works by instantiating all objects at the beginning of the scene and activate them as necessary. When a zombie is spawned, rather the creating a new one, it picks one instance that is available and make it active.
